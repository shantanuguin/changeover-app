The Problem Space

Why did we build this?

Before the portal, changeovers were chaotic. We faced systemic communication gaps that led directly to wasted time and lost production capacity.

Static Timelines

We relied entirely on a 3-month physical plan. It took hours to update and circulate, meaning critical day-to-day shifts in schedule were invisible to most departments.

Manual Paperwork

Signatures for pilot runs and approvals required physical hunting across the factory floor. The resulting delay in sign-offs cascaded into delayed production ramps.

Zero Visibility

Management had no real-time way to know which line was doing what. Analyzing historical bottlenecks required digging through filing cabinets of old checklists.

What We Did & How We Built It

A detailed breakdown of the portal's interconnected systems based on real-world engineering constraints.

Section 1

Real-Time Planning Portal

Due to the absence of real-time updates from the Planning Department, the 3-month production plan served as the primary reference for all departments. However, this document required time to be updated and circulated, and since it was typically revised only when significant timeline changes accumulated, individual style or schedule modifications were often not reflected immediately. This resulted in delays in communication and coordination between departments dependent on production planning.

To address this gap, a real-time planning portal was developed to allow the Planning Department to directly update production changeovers as they occur. The system includes a Schedule Page and a Calendar Page, enabling all departments to view and cross-verify changeover timelines in real time. If a delay occurs — for example due to recuts or unfinished line WIP — the planner can immediately update the changeover schedule, and the revised timeline becomes instantly visible to all departments through the shared calendar interface.

The portal also initiates the changeover communication workflow. When the Planning Department schedules a changeover for a production line, the planner enters the current style, upcoming style, and the planned date and time of changeover into the system. Based on this information, the system automatically schedules an email notification to relevant departments four days prior to the planned changeover, ensuring adequate preparation time while preventing an overload of emails when multiple changeovers are entered simultaneously.

If the changeover schedule is modified, the system automatically generates a follow-up email notification reflecting the updated timeline, ensuring that all departments remain informed without requiring manual intervention.

To further improve accessibility, the calendar was also deployed as an Android APK application, allowing native push notifications to be delivered directly to users' mobile devices. For iOS users, the system operates as a Progressive Web App (PWA) and uses Firebase Cloud Messaging to send browser-based notifications. This ensures that changeover updates and alerts are communicated promptly across devices.

Technical Implementation
  • Schedule & Calendar Pages: Built as standalone HTML pages that read and write changeover records directly to Firebase Firestore using the JS SDK. Updates sync instantly to all clients via onSnapshot listeners—no page refresh needed.
  • Automated Emails: The Node.js + Express backend uses Nodemailer with dual SMTP accounts. When a changeover is scheduled, the frontend calculates the date 4 days prior to trigger the backend endpoint with formatted HTML templates. Includes Multer for 25MB attachment buffer logic.
  • Push Notifications: Delivered natively as an Android APK (via TWA) and progressive web app. Service Workers cache offline assets and intercept Firebase Cloud Messaging (FCM) push alerts, using the Firebase Admin SDK.
  • Hosting: The entire application is deployed on Vercel. Static HTML pages are served directly, while the Express backend is routed through Vercel serverless functions.
Section 2

Department-Wise Checklists

In addition to the master monitoring system, department-wise checklists are provided to each department through a secure login with department-specific credentials. Departments can identify their upcoming changeovers either through the automated email notifications or by referring to the Changeover Calendar within the system. Once identified, departments can access the relevant checklist and update the status of individual checklist items. They also have the option to include remarks or observations for specific items, which can later be referenced during the changeover review discussions.

This cross-department checklist framework ensures continuous visibility of preparation status and reinforces the responsibility of each department to complete their prerequisites in advance, thereby minimizing the risk of delays during the changeover process.

Furthermore, the checklist data serves as the foundation for a Changeover Review Meeting conducted one day prior to the scheduled changeover. During this meeting, representatives from relevant departments review the checklist completion status and discuss operational aspects such as production logistics, quality-critical points, special machine requirements, line WIP status for the outgoing style, and the anticipated ramp-up for the new style.

Based on the status and comments in the checklist, a Changeover Status Report is compiled. This report highlights any pending tasks or outstanding issues that require immediate action to ensure a smooth transition. This mechanism helps to proactively identify and resolve bottlenecks before the actual changeover begins.

Technical Implementation
  • Two-Tier Checklist Architecture: The Master Checklist provides an admin-facing aggregated view, whilst the Department Checklist presents individual departments only with tasks related explicitly to their designated credential-based silos.
  • Authentication: Utilizes Firebase Auth combined with a client-side password gate. The separation of access without requiring full user account management keeps onboarding seamless.
  • Real-Time Sync: Checklist toggles map to specific document fields globally within a given changeover block in Firestore via doc().update(). Listeners echo all mutations immediately on all open browser threads.
  • Changeover Status Report: Generated dynamically by querying Firestore, computing department-level task ratios into an HTML summary capable of being natively attached via email endpoints.

Section 3

Digital Signature System

A Digital Signature system has been developed to enable key personnel from various departments to sign off on specific process stages during a production changeover. This system eliminates the need for manual paperwork and physical handoffs, reducing delays and enhancing process transparency.

The digital signature workflow is implemented for several critical stages of the changeover process, including:

  1. Pilot Run Approval: Signature from the Technical, Quality, and Production departments to signify that the pilot run has been completed and that the quality and production parameters meet the required standards.
  2. First Output Quality Check: Signature from the Quality, Technical, and Production departments confirming that the first production output from the line, following the changeover, meets the required quality specifications.
Technical Implementation
  • Signature Capture Flow: Implemented transparently within both Recorder APIs and Portal Creation pages. Actions capture the signer's context role, precise UTC timestamp, and decision flag—strictly enforcing rule constraints against repetitive signatures from the same authorized groups.
  • State Persistence: Stored immutably as highly nested object fields attached directly to respective changeover instances inside Firestore (e.g. signatures.firstOutput.production), enforcing an unalterable chain-of-custody.
  • Real-Time Visibility: A universal Live Monitor propagates these immutable transactions immediately to management dashboards in milliseconds without page refreshes.
Section 4

Reporting System

A Reporting System has been developed within the Production Changeover Portal to enable consistent monitoring and evaluation of changeover performance. This system provides visibility into the overall prep and sign-off status of changeovers, helping all departments identify areas for improvement and optimize the changeover process.

The portal generates a variety of reports, including:

  1. Pilot Run Approval Status: A summary of all changeovers where the pilot run has been approved, which helps the Management identify and address any bottlenecks in the pilot run process.
  2. First Output Quality Check Status: A summary of all changeovers where the first output quality check has been approved. This report helps management identify and address any quality-related issues during the changeover process.

By leveraging these reports, management can identify trends, monitor performance against targets, and make informed decisions regarding process improvements. The Portal's reporting system ensures that all stakeholders have access to real-time information.

Technical Implementation
  • Reports DOM Engine: Features a tiny footprint (101KB payload) that filters complex NoSQL query streams, iterating client-side variables logically rather than employing slow server queries. Provides tabular HTML DOM manipulation intuitively.
  • Smart Filtering algorithms: Highlights overdue occurrences specifically using condition mapping (signatures.* exists boolean parameters) alerting delays visually immediately depending on predefined system tolerances.
  • Email & Export Options: Self-referencing the client UI HTML generation as a scalable string that dynamically pipes over REST endpoints straight into structured automated Nodemailer PDFs / Spreadsheets.

Section 5

Continuous Improvement & Analytics

A Continuous Improvement and Analytics Dashboard has been integrated into the changeover system to provide high-level insights into operational efficiency. This dashboard allows the Management team and Department Heads to visualize trends and identify persistent issues across multiple changeovers.

Key metrics tracked in the dashboard include:

  • Total Preparation Lead Time: The time between the start of preparation (triggered by the planning update) and the actual changeover commencement.
  • Checklist Compliance: The percentage of pre-required items completed by each department before the deadline.
  • Ramp-up Efficiency: A comparison between planned and actual production rates during the initial hours of a new style.
  • Delay Attribution: Analysis of reasons for changeover delays (e.g., machine downtime, material shortage, pilot run delay), categorized by department and style category.

These analytics enable data-driven discussions during monthly review meetings, fostering a culture of accountability and continuous process refinement.

Technical Implementation
  • Dashboard Metrics Engine: Connects to Firestore to compute complex metrics in real-time on the client side without throttling the primary UI. Computes metrics (like Compliance and Lead Time distributions) gracefully against customized date buckets.
  • Ramp-up Data Layer: Recipient Recorder Pages aggregate high-frequency bi-hourly operations records and correlates these raw integers directly to standard capacity models stored within the global Style Repository arrays.
  • Planning Logic Module: Implements an incredibly dense 98KB functional code module governing dynamic schedule constraints and overlap prevention math, rendering natively into the Live Monitors to flag pipeline issues prior to them resulting in losses.
Section 6

Historical Knowledge Base

A Historical Knowledge Base has been created to store all information related to previous changeovers. This knowledge base serves as a valuable resource for departments, allowing them to:

  • Reference past checklist items and remarks: Identify recurring issues and best practices from previous changeovers of the same style or department.
  • Review historical changeover schedules: Gain insights into typical changeover durations and identify areas for potential optimization.
  • Access past changeover reports and status reports: Review historical performance and identify trends over time.

By leveraging this historical data, departments can proactively address potential challenges, optimize their changeover preparations, and contribute to the overall efficiency of the production process. The knowledge base is accessible through the Changeover Portal, ensuring all relevant stakeholders have access to the information they need.

Technical Implementation
  • NoSQL Knowledge Architecture: Because Firestore leverages non-relational Document parameters, individual occurrences map flawlessly natively retaining contextual signatures, dates, timestamps, checklist checks, and associated style arrays permanently into a single scalable object wrapper.
  • Style Repository Implementation: Houses isolated Style definitions specifying garment complexities, efficiency variables, handling attributes mapping directly backwards generating references per each QCO history, populating "Unique Style" tracking.
  • Remark Architcture: Natural language "Remarks" and Observations submitted ad-hoc coexist alongside strict boolean values, indexing successfully across historical timeframe queries enabling pattern analyses natively.

Security & Access Control

The Express backend enforces a dynamic CORS whitelist, permitting operations only explicitly inside configured Vercel origins mapping to production servers.

Email vectors are rate-limited utilizing express-rate-limit preventing spam abuse. Firebase SDK Rules enforce role mapping restricting untrusted mutations on critical Firestore paths seamlessly.

UI/UX & Glassmorphism

Implemented extensively with frosted glass layers utilizing TailwindCSS backdrop filters cascading against animated ambient backgrounds mapped carefully under high z-indexes.

Components are strictly mobile-first responsive handling interactions via minimum hit areas mapping. Render cycles optimize for touch via dynamic CSS safe-area-insets ensuring notched screens respond correctly.

PWA & Offline Capability

The native layout ships a comprehensive manifest.json alongside maskable icons providing standard Standalone display models pushing users into OS-level application behaviors instantly upon installation flags.

Service workers map strictly separated offline configurations, rendering cached UI layers via stale-while-revalidate protocols intercepting APIs purely for background notifications exclusively instead of blocking standard data cycles.
