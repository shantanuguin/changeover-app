The Problem Space

Why did we build this?

Before the portal, changeovers were chaotic. We faced systemic communication gaps that led directly to wasted time and lost production capacity.

Static Planning Timelines

We relied entirely on a physical 3-month plan. Updates took too long to circulate, leaving departments entirely dependent on an outdated schedule when changes occurred.

Uncoordinated Prerequisites

Critical prerequisites required from multiple departments were frequently missed due to a lack of shared monitoring, which directly resulted in delayed changeover executions.

Inaccurate Machine Planning

Because costing Operation Bulletins rarely reflected actual floor configurations, requested machines were often inaccurate, leading to missing equipment or unused excess machines.

Manual & Isolated Tracking

Recorded via pen and paper by a single executive, changeover visibility remained stubbornly restricted. We could not properly track break times or reliably identify which department caused delays.

Scattered & Siloed Data

Reporting lived inside disconnected Excel sheets. Cross-line visibility was non-existent. Without a centralized view, analyzing broader performance trends across units was impossible.

Loss of Historical Insight

Because we failed to build a centralized style repository, operational knowledge from past changeovers was routinely lost, making it difficult to learn from recurring bottlenecks.

What We Did & How We Built It

A detailed breakdown of the portal's interconnected systems based on real-world engineering constraints.

Section 1

Real-Time Planning Portal

Due to the absence of real-time updates from the Planning Department, the 3-month production plan served as the primary reference for all departments. However, this document required time to be updated and circulated, and since it was typically revised only when significant timeline changes accumulated, individual style or schedule modifications were often not reflected immediately. This resulted in delays in communication and coordination between departments dependent on production planning.

To address this gap, a real-time planning portal was developed to allow the Planning Department to directly update production changeovers as they occur. The system includes a Schedule Page and a Calendar Page, enabling all departments to view and cross-verify changeover timelines in real time. If a delay occurs — for example due to recuts or unfinished line WIP — the planner can immediately update the changeover schedule, and the revised timeline becomes instantly visible to all departments through the shared calendar interface.

The portal also initiates the changeover communication workflow. When the Planning Department schedules a changeover for a production line, the planner enters the current style, upcoming style, and the planned date and time of changeover into the system. Based on this information, the system automatically schedules an email notification to relevant departments four days prior to the planned changeover, ensuring adequate preparation time while preventing an overload of emails when multiple changeovers are entered simultaneously.

If the changeover schedule is modified, the system automatically generates a follow-up email notification reflecting the updated timeline, ensuring that all departments remain informed without requiring manual intervention.

To further improve accessibility, the calendar was also deployed as an Android APK application, allowing native push notifications to be delivered directly to users' mobile devices. For iOS users, the system operates as a Progressive Web App (PWA) and uses Firebase Cloud Messaging to send browser-based notifications. This ensures that changeover updates and alerts are communicated promptly across devices.

Technical Implementation
  • Schedule & Calendar Pages: Built as standalone HTML pages that read and write changeover records directly to Firebase Firestore using the JS SDK. Updates sync instantly to all clients via onSnapshot listeners—no page refresh needed.
  • Automated Emails: The Node.js + Express backend uses Nodemailer with dual SMTP accounts. When a changeover is scheduled, the frontend calculates the date 4 days prior to trigger the backend endpoint with formatted HTML templates. Includes Multer for 25MB attachment buffer logic.
  • Push Notifications: Delivered natively as an Android APK (via TWA) and progressive web app. Service Workers cache offline assets and intercept Firebase Cloud Messaging (FCM) push alerts, using the Firebase Admin SDK.
  • Hosting: The entire application is deployed on Vercel. Static HTML pages are served directly, while the Express backend is routed through Vercel serverless functions.
Section 2

Changeover Checklist System

After a changeover is scheduled by the Planning Department, it is essential to ensure that all prerequisite requirements across departments are completed before the changeover begins. If critical prerequisites are missing, the changeover process may face delays during execution. To address this issue, a structured Changeover Checklist System was developed to monitor and ensure preparedness across all departments.

The system consists of two types of checklists: a Master Checklist and Department-wise Checklists.

The Master Checklist is managed by the Manufacturing Excellence (MEX) team. If the cumulative completion status falls below 90%, the system records the progress of each department and automatically generates an alert email 42 hours before the scheduled changeover. This email summarizes the overall completion percentage and highlights the progress status of each department.

In addition to the master monitoring system, department-wise checklists are provided to each department through a secure login with department-specific credentials. Departments can access the relevant checklist and update the status of individual checklist items. The checklist data also serves as the foundation for a Changeover Review Meeting conducted one day prior to the scheduled changeover.

Technical Implementation
  • Two-Tier Checklist Architecture: The Master Checklist provides an admin-facing aggregated view, whilst the Department Checklist presents individual departments only with tasks related explicitly to their designated credential-based silos.
  • Authentication: Utilizes Firebase Auth combined with a client-side password gate. The separation of access without requiring full user account management keeps onboarding seamless.
  • Real-Time Sync: Checklist toggles map to specific document fields globally within a given changeover block in Firestore via doc().update(). Listeners echo all mutations immediately on all open browser threads.
  • Changeover Status Report: Generated dynamically by querying Firestore, computing department-level task ratios into an HTML summary capable of being natively attached via email endpoints.

Section 3

IE Preparations & Machine Requirement List

Parallel to the completion of departmental checklists, the Industrial Engineering (IE) department prepares for the upcoming changeover by uploading the Operation Bulletins (OBs) of both the current style and the upcoming style into the creation portal. The system extracts key parameters from the OB, including the list of operations, standard minute values, machines allocated, and the number of operators and machines required.

Using this information, the system performs a comparative analysis to generate a machine requirement list for the changeover. This identifies which machines are already present on the line and which additional machines will be required.

Initially, the costing OB was used to estimate machine requirements. However, costing OBs often do not reflect the latest production-level modifications. By shifting to the updated production OB, the system ensures a more accurate representation of the actual operations being executed on the line.

Once the machine comparison analysis is generated, the IE department circulates the machine requirement report via email to all stakeholders. This preparation reduces the time required for machine setup on the production line and helps minimize delays during the transition.

Technical Implementation
  • CSV Parsing: Operation Bulletins (OBs) are parsed client-side using the PapaParse library for robust handling of standard data grids.
  • Comparison Logic: Custom JavaScript iterates through required machines per operation, mapping them against the active line's equipment to compute the net delta (shortage/surplus).
  • Data Storage: The computed machine requirement dataset is saved immutably directly to the specific QCO document inside the Firestore database.
Section 4

Digital Changeover Recording System

After the changeover is scheduled, the relevant OBs are uploaded, and departmental checklists are followed up, the next stage involves recording and monitoring the changeover process during execution.

Previously, changeovers were recorded manually. This approach created limitations. The visibility of the changeover was restricted, and it was difficult to accurately track which department contributed to delays, the duration of each activity, or the financial impact of the lost time.

To overcome these limitations, a digital changeover recording system was developed within the PWA. The system allows changeovers to be recorded through a simplified interface consisting of four primary inputs: Start, Stop, Department Responsible, and Remarks.

The system includes three different recording modes to accommodate various operational scenarios:

  • Admin Recorder: Includes the standard recording interface with editing capabilities.
  • Field Recorder: Same recording interface without editing permissions.
  • Instant Recorder: Designed for unplanned or unscheduled changeovers.

The system also features a Live Changeover Monitor, which securely manages and relays real-time process states to all corresponding dashboards simultaneously.

Technical Implementation
  • Role-Based Access Control (RBAC): The three recording modes (Admin, Field, Instant) are securely gated via frontend application states ensuring correct data provenance.
  • Synchronized Timing: Timestamps are captured via Date.now() and synced precisely with firebase.firestore.FieldValue.serverTimestamp() to prevent local clock manipulation.
  • Event Transmissions: Intercept listeners bind to active Firebase documents pushing micro-updates as arrays without overriding complete data structures concurrently.

Section 5

Centralized Reporting System

Earlier, the reporting system relied on individual Excel sheets that primarily generated Loss Time Breakdown and Department-wise Loss Analysis for each recorded changeover. While these reports provided useful insights, they were limited to the specific Excel file. This meant there was no centralized view of changeover performance across multiple lines.

To address this limitation, a centralized reporting system was developed within the digital changeover management platform. This system consolidates data from all recorded changeovers and generates comprehensive analytical reports that provide both operational and financial insights.

The centralized dashboard enables users to view and analyze multiple parameters, including Loss Time Breakdown, Department-wise Loss Distribution, Total Cost Impact with cost component breakdowns, SMV Multiple Analysis, and Cost Parameters associated with changeover delays.

In addition, the system highlights Top 5 Loss Time Contributors, allowing management to quickly identify the most significant sources of inefficiency. By consolidating all changeover data into a single reporting interface, the system enables management to track performance trends, compare line efficiencies, and make more informed decisions.

Technical Implementation
  • Data Visualization: Analytics are rendered using Chart.js for visual data representation (Loss Time Breakdown, Department Loss) inside responsive Glassmorphic dashboard containers.
  • Real-time Computation: The SMV Multiple and cost calculations are computed dynamically in JavaScript by factoring the total recorded time against the Target SMV and the user-input Cost/Manpower parameters.
  • Cross-collection Queries: Queries aggregate multi-line data effortlessly utilizing Firestore's powerful compound capabilities to filter across periods and lines simultaneously.
Section 6

Style Repository & Planning Sheet Module

In addition to the operational modules, the system also includes a Style Repository and a Planning Sheet Module to support data organization and future reference.

The Style Repository functions as a centralized database where every unique style handled by the factory is recorded. This repository allows users to retrieve historical information related to past changeovers, including changeover overviews, style comparisons, daily efficiency data, and machine analysis reports. By maintaining a structured archive, the system enables departments to review previous transitions and use these insights to better plan future changeovers.

Complementing this repository is the Planning Sheet Module, which integrates the factory’s 90-day production planning sheet into the system. Based on this planning data, the module analyzes the production sequence and provides automated suggestions on the Schedule Page regarding potential changeover timings. The Planning Department can then review these recommendations and either proceed with the suggested schedule or manually create a changeover.

This integration between the planning sheet and the scheduling interface helps streamline the transition from long-term production planning to actionable changeover scheduling, improving planning accuracy across departments.

Technical Implementation
  • Integration Mechanism: The Planning Sheet integration reads from an exported data file (often Excel/CSV transformed into JSON) and populates a Select2 dropdown interface to reduce typing overhead.
  • Index-Based Search: The style archival system querying relies on structured Firestore indices (e.g., qcoNumber, currentStyle, upcomingStyle) executing fast prefix and combination searches globally.
  • Data Architecture: Natural language remarks and relational mappings are stored within scalable document blocks, isolating complex nested parameters to circumvent the typical constraints of NoSQL document scaling.

Technical Implementation

Application Architecture

A decoupled web ecosystem optimized for strict real-time syncing, complex temporal logic, and resilient data processing.

Frontend Client

  • Data Ingestion Engine: Employs SheetJS (XLSX) and PapaParse for complex client-side parsing of Operation Bulletins, extracting metadata via regex and mapping structural markers like EOL-TB.
  • Progressive Web App (PWA): Fully installable via manifest.json, utilizing Service Workers for stale-while-revalidate caching and a Trusted Web Activity (TWA) wrapper.
  • Responsive Glassmorphism: Constructed with Vanilla JS and TailwindCSS, featuring backdrop-filter, touch-action manipulations, and GSAP timelines for an app-like experience.

Backend Serverless

  • Real-Time Data Sync: Leverages Firebase Firestore's native WebSockets for instant state replication with built-in document locking and migration logic during schedule updates.
  • Communications Stack: Features a specialized Dual-SMTP failover system using Nodemailer for automated HTML emails, and FCM (Firebase Cloud Messaging) for real-time mobile push notifications.
  • Vercel & Express: The Node.js application is shipped on Vercel, protected by express-rate-limit and custom CORS mapping, handling robust API endpoints.
